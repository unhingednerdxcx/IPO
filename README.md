# About

Sequender is a Time management tool that both makes your time productive while saving the Earth ⏰🌏! You can manage your time by setting up **routines** and **tasks** with **deadlines**. The apps polished **UI** and sophisticated **UX** makes the experience of using the app both clean and swift. You can set various colors and the app itself is fully **Cross-platform**, meaning you can run it on **MacOS(Darwin), Windows, Linux**, etc. (as long as you have installed a browser and some other dependencies (check the *dependencies* section for more information about the dependencies this app requires))🖥️. This is thanks to taking the **web-app** approach. The daily challanges are **sustainable** tasks (like throwing out trash properly) and if the user does them, they gain **XP** which increases their **level**. This all encourages the user to do the sustainable tasks.

# Features

Sequender combines multiple key UX principles to create an amazing user experience. Upon opening the app for the first time, You will be presented with a login screen. You can choose to log-in using **Google authentication** or **E-mail** 📬. After signing in, you can make new tasks, routine, and even do challanges. These challanges are automatically generated by `generator.py` and these challanges are stored in my apps firestore, specifically, at `Tasks/Tasks` (which also contain other data but we will cover that in the *Technical details* section). `generator.py` automatically runs every midnight (thanks to Github actions and also check `.github/workflows/schedual.yaml` to learn more about the automation). Going back to the tasks, you can catagorize them and the ap will automatically tell you which tasks you have to do today 📋. The relatively simple yet elegant UI keeps you focused on your tasks. **Google**'s material icons and fonts were used to create a native-like UI

# Technical details

## Frontend
The frontend was built with HTML/CSS/TS and JS. I imported **Google** material icons and sour gummy font and used them throughout the UI. I chose to not use any frameworks like vue, or react to reduce bundle size and for future flexibility. I used chart.js(**CDN** style) to generate graphs quickly. I chose to use **Typescript** because it improves maintainability, it is more scaleable, and finds error before runtime to ensure there are no underlying bugs🐛. However, you will also notice that I also used JS. I chose to use ***Javascript*** strictly for **firebase** related commands. This is because Javascript already has a well documented SDK support while also reducing boilerplate. In the future, if my firebase does get even more complexer, TypeScript would bring unnecessary complexity in the form of additional type definitions, boilerplate and setup, while providing only marginal benefits for the scope of this project. All the compiled Typescript and Javascript files are in the dist folder 📁.

## Backend
The backend is fully made with python. Python was best suited for the backend because of its incredible data manupilation tools, so in the future, if I would like to add even more features, python would make it easier to do. In python, the main module I am using is **Eel** (⚠️ Even though its deprecated). I chose to use **Eel** instead of other libraries like **FastAPI** because Eel is more lightweight and is more flexable. due to its simple setup, it allows rapid prototype. That being said, later in the future if my scales beyond, I will adventually use **FastAPI** or **Electron**. Eel more like a RPC rather than an IPC, because it bridges functions. I would have prefered to move the firebase functions from js to python but firebase provides only the admin SDK for python and leaking my service-account.json would be extreemly risky 🔑. Hence, I didn't move the logic. The backend deals with all the json fetching and everything else. For data handling I used **pandas** and for search options, I used **rapidfuzz**. I like using a 'file-pathy' way to transfer data (so if taskA is stored in subcatA which is stored in catA, i would send it like:- `"catA/subcatA/taskA"`)

## Database
I used **Firebase** to store data. We will go over a couple sections to cover the database.

### Rules
My firebase rules are unfortunally not public as it contains keys that are properitory and leaking them can endanger my database. However, I will proide an overview of the rules. In the users/ collection, users will have their own document and **USERS CAN ONLY READ THEIR OWN DOCUMENT**. The Tasks/ collection stores all the challange details and can be read by anyone (even if not signed in) **BUT CANT BE WRITTEN BY ANYONE(the automation section will explain how I still update the challanges each day even though its hardcoded to not let anyone write to the document)**

### User collection
Each users document consists of the keys `level`, `maxXp`, `xp`, `TasksDone`, and their types are `i64`, `i64`, `i64`, `bool arr[7]` (respectively). The level stores the level user is at, maxXp stores how much xp the user needs to get to before they can increase their level (for example if the maxXp is 100, and the users level is at 0, and the xp is at 99, and then do a task, causing their xp to go beyond 100, maxXp will be increased and the level would be incremented by 1)

### Tasks document
The Tasks document consists of 2 keys, `Tasks` and `XP` and their types are `str arr[7]`, `int arr[7]`. The Tasks key stores the tasks the user must do and the XP key stores the xp that will awarded if they do the task corresponding (For example if they do Tasks[0], their xp will be incremented by XP[0]).
